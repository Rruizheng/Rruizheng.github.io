# Babel基础

# Babel的概念

  • 概念：JS的编译器
  • 设计理念
    • 可插拔
    • 可调试
    • 基于协定
  • 架构：Babel 是一个使用 Lerna 构建的 Monorepo 风格的仓库，在其./packages目录下有 140 多个包

# 需要Babel的原因

- 向后兼容：宿主环境（浏览器、NodeJs）无法第一时间支持最新的语言特性，需要对语言特性进行编译降级，以适配不同的环境
- 将各种前端框架编译成JavaScript代码

# Babel的功能

- 语法转换，指高级语法的降级
- Polyfill特性的实现和接入（通过引入第三方 polyfill 模块，例如 core-js）
- 源码转换（codemods），例如JSX

# Babel的工作原理

1.解析：Babel首先会对JavaScript代码进行解析，生成抽象语法树（AST），以便更好地理解代码的结构和含义。 2.转换：Babel会对AST进行转换，将其中不被目标环境支持或需要变换的代码结构，例如新语法特色或全局API引入，转换成在目标环境中可以正常运行的代码。 3.生成：最终，Babel会利用改造过的AST，生成符合目标环境要求的JavaScript代码。

# 配置方式

具体配置可以在 babel.config.js 或 .babelrc 文件中配置，也可以在package.json之中使用babel关键字直接编写。

其中 babel.config.js 是一个 Node 模块文件，而 .babelrc 是 JSON 格式的配置文件。

  • babel.config.js
module.exports = {
    presets: ["@babel/preset-env"],
    plugins: [
        "@babel/plugin-transform-arrow-functions"
    ],
};

# 配置内容

Babel 常用的配置内容

1. presets：预设用于转换不同的 ECMAScript 版本或语法。

2. plugins：插件用于提供其他的转换功能，例如转换 react、stage-x。

3. env：为不同的环境配置不同的预设和插件，例如用不同的预设和插件在开发环境和生产环境中转换代码。

4. include/exclude：设置指定文件需要或不需要转换。

5. sourceType：指定 JavaScript 代码的来源类型是 `script` 还是 `module`。

6. ignore：指定 Babel 忽略某些文件。

7. overrides：为某些文件或文件夹设置不同的转换规则。

8. comments：是否将原始代码中的注释也输出到转换后的代码中。

9. sourceMaps：是否生成并输出源代码的 sourcemap。

10. compact：生成的代码是否忽略多余空白字符和换行符。

11. retainLines：是否在输出的代码中保留原始代码的行号。

12. minified：是否对代码进行最小化压缩。

13. copyFiles：是否在转译时将其他的文件或文件夹复制到输出的文件夹中。

14. babelrcRoots：babel 在哪些目录下寻找 .babelrc 文件。

15. overrides：为指定文件或目录设置不同的转换规则。

# 配置项合并优先级

当使用 Babel 进行代码转换时，会存在多个配置文件，例如 .babelrc，babel.config.js 等。这些配置文件的优先级和合并规则是什么呢？

# Babel配置加载顺序

加载顺序优先级：

  • 从命令行参数 --config-file 和 --extends 获取配置文件路径。

  • 如果项目根目录下存在 babel.config.js，则加载该文件。

  • 如果项目根目录下存在 .babelrc 或者 .babelrc.js 文件，则加载该文件。

  • 如果本身没有任何选项或配置，则使用默认配置。

# Babel配置项合并规则

  • 如果存在相同的配置选项，则后面的配置文件会覆盖前面的配置文件。例如，后面的 .babelrc 文件中的配置会替换前面 .babelrc 文件中的配置。
  • Presets（预设）和 plugins（插件）列表会被顺序合并，从而将所有的插件和预设组成一个列表。不能在插件中定义名称相同或名称插件，插件的配置项会始终被后面进行覆盖。
  • assumptions, parserOpts 和 generatorOpts, 对象被合并，而不是被替换。
